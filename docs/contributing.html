Contributing

All dashboard code changes start in the template; the builder and the canonical dashboard receive them. This page covers the mechanics — see Integration workflow for the full lifecycle.

Where code lives

  • DIMS_dashboard_template — the dashboard itself: frontend (js/, css/, index.html), analysis scripts (opt/step_*.py), and serve.py. Make code changes here.
  • DIMS_dashboard_builder — the no-code wizard. Changes to the wizard itself (the Flask app in app/, the UI in app/static/) are made directly in that repo.
  • DIMS_Dashboard — canonical data + the deployed site; receives template code, never authors it.

How the template lives inside the builder

The builder ships a complete copy of the template under template/ so a non-technical user needs neither git nor a network connection (acquire_template() in app/project.py copies it into each generated project, and the builder runs the project's own opt/ scripts).

That directory is a git subtree of the upstream template repo — the files are committed in the builder (offline-safe), but their history is linked upstream so updates pull in with one command.

Adding a feature to the dashboard

  1. Open a PR against the template

    Branch DIMS_dashboard_template (frontend or opt/ scripts) and open a pull request.

  2. Test it on real data

    Try the branch on DIMS_Dashboard and preview with serve.py — see Integration workflow.

  3. Merge to template/main

    This is the single bless point.

  4. Integrate into the builder

    cd DIMS_dashboard_builder
scripts/update-template.sh
# equivalently:
# git subtree pull --prefix=template \
#     https://github.com/dims-network/DIMS_dashboard_template.git main --squash

    Review the squashed merge, do a quick build to sanity-check, then git push the builder.

Existing generated projects pick up new scripts automatically: on the next build, acquire_template() refreshes a project's template code (opt/, frontend, deploy workflow) from the bundled copy while preserving the user's config.json and assets/.
