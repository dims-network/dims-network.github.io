Architecture

The DIMS system is several repositories with one source of truth for dashboard code. Everything else either contributes code back to that source or consumes it.

The source-of-truth model

DIMS_dashboard_template holds the canonical dashboard code — the frontend (js/, css/, index.html) and the analysis scripts (opt/step_*.py). Two downstreams consume it; each adds something different.

          DIMS_dashboard_template          # single source of truth — code only
            (frontend + opt/ scripts)
                     │
          ┌──────────┴───────────┐
      git merge               git subtree
          │                      │
    DIMS_Dashboard         DIMS_dashboard_builder
    code + real data,      template/ vendored inside;
    deployed to Pages      no-code wizard wraps it
RepoWhat it addsPulls template code via
DIMS_dashboard_template— (it is the source)—
DIMS_Dashboardcanonical data + the deployed GitHub Pages sitegit merge template/main
DIMS_dashboard_builderthe no-code wizardgit subtree pull (its template/ subdir)

Why two different mechanisms? In the builder the template lives in a template/ subdirectory, which is exactly what a git subtree tracks. In DIMS_Dashboard the dashboard code sits at the repo root, intermixed with data — there is no subdirectory to target — so it consumes the template as a plain upstream remote it merges from.

The golden rule

Never edit dashboard code directly in DIMS_Dashboard or the builder. All code changes originate in the template; the downstreams only ever receive. That is what keeps merges conflict-free — a downstream's data files never exist upstream, so they never collide.

Code vs. data

Each consumer cleanly separates code (received from the template) from its own data/configuration (kept locally):

  • DIMS_Dashboard — keeps assets/ and config.json; a .gitattributes rule (config.json merge=ours) auto-keeps its config on every template merge.
  • builder — its acquire_template() refreshes a generated project's template code (opt/, frontend, deploy workflow) on each build while preserving the user's config.json and assets/.

Related repositories

Beyond the dashboard family, the network includes Python/Dash viewers (bots-viewer, Ortho-viewer) and study-specific forks (e.g. DIMS_Dashboard_Ortho). See the repository overview on the home page.
