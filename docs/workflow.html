Integration workflow

There is one source of truth for dashboard code — the template. Features are developed there, validated on real data, blessed once, and then pulled into the downstreams.

RepoRoleGets template code via
DIMS_dashboard_templatecanonical code (frontend + opt/ scripts)— (the source)
DIMS_Dashboardcanonical data + deployed site + integration test bedgit merge template/main
DIMS_dashboard_builderthe no-code wizardgit subtree pull
Golden rule: never edit dashboard code directly in DIMS_Dashboard or the builder. All code changes originate in the template; the downstreams only receive. That keeps merges conflict-free — a downstream's assets/ never exist upstream, and config.json is auto-kept via .gitattributes merge=ours.

Lifecycle of a feature

  1. Develop on the template

    Branch DIMS_dashboard_template and open a PR there (frontend or opt/ scripts).

  2. Integration-test on real data

    In DIMS_Dashboard, try the feature branch against the canonical data and preview locally:

    git fetch template <feature-branch>
git checkout -b test/<feature> && git merge template/<feature-branch>
python serve.py 8000        # open http://localhost:8000 and check it

    Throw the test branch away afterwards (git checkout main && git branch -D test/<feature>).

  3. Bless it

    Merge the PR into template/main — the single approval point.

  4. Propagate downstream (mechanical, no re-review)

    DIMS_Dashboard:

    scripts/update-from-template.sh   # git fetch + merge template/main
python serve.py 8000              # verify with real data
git push                          # Pages redeploys

    Builder:

    scripts/update-template.sh        # git subtree pull --prefix=template ... --squash
git push

One-time setup (per clone)

DIMS_Dashboard uses an ours merge driver so config.json auto-resolves to its own version on every template merge:

git config merge.ours.driver true

The template remote is added automatically by scripts/update-from-template.sh if missing.

Why bless before propagating? Merging the feature independently into three repos invites drift. Bless once at template/main; the downstreams only ever pull.

See also Contributing for the subtree mechanics, and Architecture for the model.
